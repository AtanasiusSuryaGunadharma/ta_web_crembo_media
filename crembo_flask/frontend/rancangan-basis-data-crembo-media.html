Rancangan Basis Data Website Crembo Media

Diagram ini disusun ulang secara logis dari file SQL dan pemakaian tabel pada app.py agar relasi utama menjadi jelas. Fokus rancangan ada pada pusat data anggota sebagai master, lalu tabel transaksi dan konten yang terhubung ke sana.

Master data: anggota Relasi banyak-ke-satu dan many-to-many Siap diekspor ke PNG/PDF
Logical ERD
anggota
  • PK: id
  • nama, username, telp, email, password
  • role, tgl_lahir, alamat, status_akun
  • inactive_from, inactive_until, inactive_type
Pusat identitas pengguna, admin, dan anggota aktif/nonaktif.
admin_module_permissions
  • PK/FK: member_id → anggota.id
  • streaming, inventaris, publikasi
  • konten, keanggotaan
Relasi 1-1 khusus akun admin.
password_reset_otps
  • FK logis: member_id → anggota.id
  • member_email, otp_hash
  • reset_token_hash, attempts
Riwayat OTP reset password per anggota.
notifications
  • Relasi logis: target_user_id di data JSON
  • type, title, body, url
  • target_role
Notifikasi umum dan personal.
notification_reads
  • PK gabungan: notification_id + user_key
  • read_at
Menandai notifikasi sudah dibaca.
notification_email_deliveries
  • FK logis: notification_id → notifications.id
  • recipient_user_id, recipient_email
  • status, sent_at
Audit pengiriman email notifikasi.
news_categories
  • PK: id
  • name, slug, description
  • order_index
Kategori pengumuman / berita.
news
  • PK: id
  • title, slug, content, summary
  • thumbnails, attachments, status
Konten publik utama yang bisa multi-kategori.
news_category_mapping
  • FK: news_id → news.id
  • FK: category_id → news_categories.id
  • many-to-many news - kategori
Tabel penghubung berita dan kategori.
agendas
  • PK: id
  • title, description, start/end date
  • location, registration_link, attachments
Agenda kegiatan gereja.
registration_forms
  • PK: id
  • FK logis: created_by → anggota.id
  • target, visibility, quota, fields_json
Form pendaftaran publik / internal.
registration_form_submissions
  • FK: form_id → registration_forms.id
  • FK logis: submitter_user_id → anggota.id
  • submitter_key, answers_json
Hasil pengisian form.
organization_profiles
  • PK: id
  • title, description, attachment_url
  • is_visible, order_index
Konten profil organisasi.
inventory_categories
  • PK: id
  • category_name, slug
Master kategori barang.
inventory_items
  • FK: category_id → inventory_categories.id
  • item_name, code, stock, unit_status
  • photos, unit_details
Data barang inventaris.
loan_requests
  • FK logis: member_id → anggota.id
  • barang_id, barang_name, jumlah
  • status, approved_by, pickup_info, return_info
Transaksi peminjaman / pengembalian.
form_kerusakan_barang
  • FK logis: member_id → anggota.id
  • barang_id, barang_name
  • tingkat_kerusakan, status, foto_kerusakan
Laporan kerusakan barang.
streaming_roles
  • PK: id
  • role_name, order_index
Master peran untuk jadwal streaming.
streaming_weekly_config
  • PK: id
  • day_name, start_time, mass_name
Template jadwal misa mingguan.
streaming_assignments
  • FK logis: member_id → anggota.id
  • schedule_date, schedule_time, role_name
  • request_source
Penugasan misa biasa.
task_exchange_requests
  • FK logis: requester_id → anggota.id
  • FK logis: target_user_id → anggota.id
  • my_assignment_id, my_misa_id, my_role_id
Permintaan tukar tugas.
task_cancellations
  • FK logis: member_id → anggota.id
  • assignment_id, role_id, misa_id
  • status, note, request_source
Riwayat pembatalan tugas.
streaming_evaluations
  • FK logis: evaluator_id → anggota.id
  • schedule_kind, schedule_key
  • staff_json, checklist_json, final_note
Penilaian hasil pelayanan streaming.
misa_besar
  • PK: id
  • misa_name, misa_date, misa_time
  • allow_member_request, status
Master jadwal misa besar.
misa_besar_names
  • FK: misa_id → misa_besar.id
  • role_name, required_count, order_index
Daftar role pada satu misa besar.
misa_besar_assignments
  • FK: role_id → misa_besar_names.id
  • FK: member_id → anggota.id
  • request_source, created_at
Relasi penugasan misa besar.
sertifikat_config
  • Relasi logis: dipakai saat generate sertifikat anggota
  • romo_name, ketua_name, pembina_name
  • romo_sign_url, ketua_sign_url, pembina_sign_url
Konfigurasi tanda tangan sertifikat.
Inti perbaikan rancangan: tabel pusat dibuat jelas di anggota, lalu tabel-tabel yang sebelumnya hanya menyimpan id teks atau JSON disarankan memakai relasi FK yang eksplisit. Dengan begitu, laporan lebih logis dan hubungan data lebih mudah dibaca.
Prinsip desain: master data, transaksi, dan konten dipisah. Tabel jembatan dipakai untuk relasi many-to-many, misalnya berita dan kategori.
Catatan akademik: kolom seperti created_by, member_id, admin_id, dan submitter_user_id idealnya mengarah ke anggota.id agar integritas referensial terjaga.

Relasi utama

anggota menjadi pusat data dan dihubungkan ke tabel transaksi seperti loan_requests, form_kerusakan_barang, password_reset_otps, serta admin_module_permissions.

Konten publik

news, news_categories, agendas, registration_forms, dan registration_form_submissions membentuk modul informasi dan pendaftaran.

Operasional gereja

streaming_assignments, misa_besar, misa_besar_names, misa_besar_assignments, task_cancellations, dan task_exchange_requests menangani jadwal serta tugas pelayanan.
